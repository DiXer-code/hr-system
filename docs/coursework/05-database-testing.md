# Тестування БД

## 1. Структурне тестування

Потрібно перевірити:

- наявність усіх таблиць;
- коректність первинних і зовнішніх ключів;
- унікальність `inn`;
- коректність типів даних;
- наявність індексів для звітних вибірок.

Це перевіряється через:

- `schema.sql`;
- запуск застосунку;
- інтеграційні тести;
- ручні SQL-перевірки.

## 2. Функціональне тестування

Автоматизовані інтеграційні тести покривають:

- створення працівника та запису в `JobHistory`;
- переведення працівника між департаментами/посадами;
- звільнення працівника;
- заборону видалення працівника при наявності `Vacation`;
- rollback при порушенні унікальності `inn`.

Тести розташовані в:

- `src/test/java/com/example/hrsystem/EmployeeServiceIntegrationTest.java`

## 3. ACID-перевірки

### Atomicity

Якщо транзакція збереження працівника падає через дубльований `inn`, жодна частина запису не повинна залишитися в БД.

### Consistency

Система не дозволяє:

- зберегти посаду, що не належить департаменту працівника;
- видалити працівника, коли існують пов'язані табелі або відпустки;
- видалити департамент із працівниками або позиціями.

### Isolation

Паралельні операції мають ізолюватися на рівні транзакцій СУБД. Для демонстрації можна запускати окремі сесії MySQL зі сценаріями з `acid_checks.sql`.

### Durability

Після успішного commit дані про працівника, історію та документи зберігаються в БД і доступні після перезапуску застосунку.
