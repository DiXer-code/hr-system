# Концептуальна, логічна та фізична моделі БД HR System

## 1. Призначення бази даних

База даних проєкту `HR System` призначена для автоматизації кадрового обліку в компанії. Вона повинна забезпечувати:

- зберігання особових карток працівників;
- прив'язку працівника до поточного департаменту і поточної посади;
- ведення кадрової історії;
- зберігання службових документів;
- облік відпусток і табелів;
- зберігання відомостей про освіту;
- авторизацію через користувачів і ролі.

У моделі центральною сутністю є `Employee`, а решта сутностей або описують його поточний стан, або фіксують допоміжні, історичні чи адміністративні дані.

## 2. Концептуальна модель

### 2.1. Основні сутності предметної області

| Сутність | Призначення | Тип сутності |
| --- | --- | --- |
| `Employee` | Особова картка співробітника | Центральна |
| `Department` | Організаційний підрозділ компанії | Довідник |
| `Position` | Посада в межах департаменту | Довідник |
| `JobHistory` | Журнал кадрових подій | Історична |
| `Document` | Накази та інші документи працівника | Підлегла |
| `Vacation` | Періоди відпусток | Операційна |
| `Timesheet` | Денний облік відпрацьованих годин | Операційна |
| `Education` | Дані про освіту працівника | Підлегла |
| `Role` | Роль доступу в системі | Адміністративна |
| `User` | Обліковий запис для входу | Адміністративна |

### 2.2. Концептуальні зв'язки між сутностями

| Зв'язок | Кратність | Зміст зв'язку |
| --- | --- | --- |
| `Department - Position` | `1 : 0..*` | Один департамент містить багато посад |
| `Department - Employee` | `1 : 0..*` | У департаменті може працювати багато працівників |
| `Position - Employee` | `1 : 0..*` | Одна посада може бути призначена багатьом працівникам |
| `Employee - JobHistory` | `1 : 0..*` | Один працівник має багато кадрових подій |
| `Employee - Document` | `1 : 0..*` | Один працівник має багато документів |
| `Employee - Vacation` | `1 : 0..*` | Один працівник має багато відпусток |
| `Employee - Timesheet` | `1 : 0..*` | Один працівник має багато табельних записів |
| `Employee - Education` | `1 : 0..*` | Один працівник може мати кілька записів про освіту |
| `Role - User` | `1 : 0..*` | Одна роль може бути призначена багатьом користувачам |

### 2.3. Концептуальна інтерпретація моделі

1. Працівник є головним бізнес-об'єктом, навколо якого побудована система.
2. `Department` і `Position` задають організаційну структуру підприємства.
3. `JobHistory` зберігає не поточний стан, а послідовність кадрових подій у часі.
4. `Document`, `Vacation`, `Timesheet`, `Education` є підлеглими сутностями, які деталізують картку працівника.
5. `Role` і `User` реалізують окремий контур безпеки та не змішуються з кадровими даними.

## 3. Логічна модель БД

Логічна модель описує таблиці, ключі та зв'язки між ними незалежно від конкретної СУБД. Для HR System логічна модель базується на реляційному підході.

### 3.1. Умовні позначення

- `PK` — первинний ключ;
- `FK` — зовнішній ключ;
- `AK` — альтернативний ключ або унікальний атрибут;
- `NULL` — атрибут може бути порожнім;
- `NOT NULL` — атрибут є обов'язковим.

### 3.3. Деталізація зовнішніх ключів

| Код зв'язку | Дочірня таблиця | Поле FK | Батьківська таблиця | Зміст |

| `L3` | `Employee` | `current_position_id` | `Position` | Поточна посада працівника |

### 3.4. Бізнес-правила цілісності, закладені в логічну модель

1. Якщо вибрано і департамент, і посаду працівника, посада повинна належати саме цьому департаменту.
2. Працівник не може мати два поточних департаменти або дві поточні посади одночасно.
3. `inn` працівника повинен бути унікальним.
4. Видалення департаменту заборонене, якщо з ним пов'язані працівники або посади.
5. Видалення працівника заборонене, якщо існують табелі або відпустки.
6. Кожна подія `ПРИЙНЯТТЯ`, `ПЕРЕВЕДЕННЯ`, `ЗВІЛЬНЕННЯ` повинна відображатися в `JobHistory`.
7. Історичні дані не повинні перезаписувати поточний стан працівника.

### 3.5. Нормалізація логічної моделі

#### Перша нормальна форма (1NF)

Всі таблиці містять атомарні атрибути. Повторювані групи не зберігаються в одному полі. Наприклад, документи і табелі винесені в окремі таблиці.

#### Друга нормальна форма (2NF)

У таблицях із простими первинними ключами всі неключові атрибути повністю залежать від первинного ключа.

#### Третя нормальна форма (3NF)

У довідниках `Department`, `Position`, `Role` і в основних операційних таблицях неключові атрибути не залежать транзитивно один від одного. Наприклад, дані про відпустку не зберігаються в `Employee`, а винесені в окрему таблицю.

## 4. Фізична модель

Фізична модель реалізована в `MySQL` і описана у файлі [schema.sql](../../database/schema.sql).

### 4.1. Основні технічні рішення

- `AUTO_INCREMENT` для первинних ключів;
- `INT` для більшості довідників;
- `BIGINT` для сутностей із потенційно великим обсягом записів;
- `DATE` для дат кадрових подій;
- `DECIMAL(12,2)` для фінансових значень;
- `LONGBLOB` для фото та документів.

### 4.2. Фізичні обмеження та індекси

| Обмеження або індекс | Призначення |
| --- | --- |
| `PRIMARY KEY` у кожній таблиці | Унікальна ідентифікація запису |
| `FOREIGN KEY` | Підтримка посилальної цілісності |
| `UNIQUE(Employee.inn)` | Забезпечення унікальності працівника |
| `idx_employee_department` | Швидкі вибірки працівників за департаментом |
| `idx_employee_position` | Швидкі вибірки працівників за посадою |
| `idx_employee_status` | Аналітика по статусах |
| `idx_employee_hire_date` | Звіти за датою прийняття |
| `idx_employee_name` | Пошук за ПІБ |
| `idx_position_department` | Вибірки посад за департаментом |
| `idx_document_employee` | Пошук документів працівника |
| `idx_job_history_employee_dates` | Отримання історії працівника в часі |
| `idx_vacation_period` | Аналіз періодів відпусток |
| `idx_timesheet_employee_date` | Табельна аналітика по датах |

## 5. Висновок

Побудована модель є сильною для курсової роботи, тому що:

- має чітко виражену центральну сутність `Employee`;
- відокремлює довідники, історію, операційні й адміністративні дані;
- відповідає вимогам нормалізації;
- містить явні ключі, зв'язки та правила цілісності;
- може бути природно розширена патернами `Queue`, `Caching`, `Audit Log`, `Versioning`.

У результаті модель не лише описує поточний стан проєкту, а й демонструє готовність системи до масштабування, контролю змін і підвищення надійності.
