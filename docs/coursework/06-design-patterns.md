# Архітектурні патерни проєктування для HR System

## 1. Призначення документа

Цей документ пояснює, які патерни проєктування доцільно використовувати в системі `HR System`, чому саме вони потрібні, які задачі вони вирішують і як можуть бути інтегровані у Spring Boot застосунок.

Важливо розділяти два рівні:

1. **Патерни, які вже явно присутні в коді проєкту.**
2. **Патерни, які доцільно закласти в архітектуру як розширення для курсової роботи.**

Такий підхід чесний і сильний для захисту: він показує не тільки поточну реалізацію, а й зрілі архітектурні рішення.

---

## 2. Патерни, які вже присутні в поточному проєкті

### 2.1. MVC

**Ідея:** розділення на модель, контролер і представлення.

**У проєкті:**

- `controller` приймає HTTP-запити;
- `entity` та `report` представляють модель;
- `templates/*.html` відображають дані користувачу.

**Приклад:**

- `EmployeeController` формує модель сторінки;
- `employee-details.html` показує форму редагування.

### 2.2. Repository

**Ідея:** інкапсулювати доступ до БД через окремий шар репозиторіїв.

**У проєкті:**

- `EmployeeRepository`
- `DepartmentRepository`
- `PositionRepository`
- інші Spring Data JPA репозиторії

**Що це дає:**

- контролери не працюють напряму з SQL;
- запити зосереджені в окремому шарі;
- спрощується тестування.

### 2.3. Service Layer

**Ідея:** винести бізнес-логіку в окремий сервісний шар.

**У проєкті:**

- `EmployeeService` відповідає за збереження працівника, перевірку статусів, синхронізацію кадрової історії, бізнес-правила видалення.

**Що це дає:**

- логіка не дублюється в контролерах;
- транзакції описані централізовано;
- легше масштабувати функціональність.

### 2.4. DTO / Report Projection

**Ідея:** використовувати окремі структури даних для звітів.

**У проєкті:**

- `DepartmentHeadcountReport`
- `PositionStaffingReport`
- `StatusDistributionReport`
- `ReportDashboard`

**Що це дає:**

- відділення аналітичних структур від JPA-сутностей;
- зменшення зв'язності між UI і таблицями БД.

## 3. Патерн проєктування черги (Queue Pattern)

### 3.1. Проблема, яку вирішує

Не всі задачі варто виконувати синхронно в момент збереження працівника. Якщо після зміни працівника потрібно:

- надіслати повідомлення;
- оновити зовнішній реєстр;
- побудувати звіт;
- виконати архівацію документів;

то така логіка може уповільнити запит користувача.

### 3.2. Ідея патерна

Після бізнес-операції система не виконує важку пост-обробку одразу, а ставить подію в чергу. Окремий worker зчитує подію і обробляє її асинхронно.

### 3.3. Де цей патерн доречний у HR System

У цьому проєкті черга найбільш логічна для:

- післястворювальної обробки документів;
- аудиту важливих змін;
- оновлення кешів після змін довідників;
- побудови великих агрегованих звітів;
- відправки повідомлень при прийнятті або звільненні працівника.

### 3.4. Логічна модель для Queue Pattern

Технічна таблиця:

- `QueueMessage(id, event_type, aggregate_type, aggregate_id, payload_json, status, attempt_count, created_at, processed_at)`

### 3.5. Потік роботи

1. Користувач змінює працівника.
2. `EmployeeService` зберігає зміни транзакційно.
3. Після успішного збереження формується повідомлення `QueueMessage`.
4. Worker обробляє повідомлення окремо від веб-запиту.
5. Статус повідомлення змінюється на `DONE` або `FAILED`.

### 3.6. Переваги

- веб-запит відповідає швидше;
- систему легше масштабувати;
- обробка подій повторювана;
- можна контролювати помилки і кількість спроб.

### 3.7. Приклад коду

```java
public void onEmployeeUpdated(Employee employee) {
    QueueMessage message = new QueueMessage();
    message.setEventType("EMPLOYEE_UPDATED");
    message.setAggregateType("Employee");
    message.setAggregateId(employee.getId());
    message.setPayloadJson(toJson(employee));
    message.setStatus("NEW");
    queueMessageRepository.save(message);
}
```

```java
@Scheduled(fixedDelay = 5000)
public void processQueue() {
    List<QueueMessage> messages = queueMessageRepository.findTop50ByStatusOrderByCreatedAtAsc("NEW");
    for (QueueMessage message : messages) {
        handleMessage(message);
    }
}
```

### 3.8. Як це пояснювати викладачу

> Черга використовується для відокремлення основної бізнес-операції від фонових задач. Це підвищує продуктивність і дає можливість безпечно обробляти події після збереження працівника.

## 4. Патерн кешування (Caching Pattern)

### 4.1. Проблема, яку вирішує

У системі є довідники і звіти, які читаються часто, але змінюються рідко:

- список департаментів;
- список посад;
- аналітичні дашборди;
- агреговані звіти по штатному складу.

Без кешування система кожного разу виконує однакові звернення до БД.

### 4.2. Ідея патерна

Результат дорогого або часто викликаного запиту зберігається в кеші. Поки дані актуальні, наступні запити повертають результат із кешу.

### 4.3. Де цей патерн доречний у HR System

- довідник департаментів;
- довідник посад;
- звіти по штатному складу;
- статистика по статусах працівників.

### 4.4. Потік роботи

1. Користувач відкриває список департаментів.
2. Сервіс перевіряє наявність результату в кеші.
3. Якщо кеш існує, дані повертаються без звернення до БД.
4. Якщо кешу немає, виконується SQL-запит, а результат кладеться в кеш.
5. При зміні департаменту кеш інвалідовується.

### 4.5. Переваги

- швидше відкриваються довідники та звіти;
- менше навантаження на БД;
- кращий користувацький досвід.

### 4.6. Приклад коду

```java
@Service
public class DirectoryService {

    @Cacheable("departments")
    public List<Department> getDepartments() {
        return departmentRepository.findAllByOrderByNameAsc();
    }

    @Cacheable("positions")
    public List<Position> getPositions() {
        return positionRepository.findAllByOrderByNameAsc();
    }

    @CacheEvict(value = "departments", allEntries = true)
    public Department saveDepartment(Department department) {
        return departmentRepository.save(department);
    }
}
```

### 4.7. Як це пояснювати викладачу

> Кешування використовується для стабільних довідкових і звітних даних. Це зменшує кількість повторних звернень до БД і прискорює інтерфейс.

## 5. Патерн журналу аудиту (Audit Log Pattern)

### 5.1. Проблема, яку вирішує

У кадровій системі важливо знати:

- хто створив працівника;
- хто змінив статус;
- хто перевів працівника в інший департамент;
- хто видалив запис;
- коли саме це сталося.

Якщо таких даних немає, неможливо коректно аналізувати дії користувачів і розслідувати помилки.

### 5.2. Ідея патерна

Кожна важлива операція над бізнес-сутністю створює окремий запис в журналі аудиту.

### 5.3. Логічна модель для Audit Log

- `AuditLog(id, actor_username, action_type, entity_name, entity_id, old_value_json, new_value_json, created_at, ip_address)`

### 5.4. Де цей патерн доречний у HR System

- створення працівника;
- редагування картки;
- звільнення;
- видалення департаменту;
- вхід до системи;
- завантаження наказів і документів.

### 5.5. Переваги

- контроль дій користувачів;
- підвищення безпеки;
- можливість відновити хронологію подій;
- допомога при перевірці і тестуванні.

### 5.6. Приклад коду

```java
public void writeAudit(String actor,
                       String action,
                       String entityName,
                       Long entityId,
                       String oldValueJson,
                       String newValueJson) {
    AuditLog log = new AuditLog();
    log.setActorUsername(actor);
    log.setActionType(action);
    log.setEntityName(entityName);
    log.setEntityId(entityId);
    log.setOldValueJson(oldValueJson);
    log.setNewValueJson(newValueJson);
    log.setCreatedAt(LocalDateTime.now());
    auditLogRepository.save(log);
}
```

```java
public Employee saveEmployee(Employee employee) {
    String before = loadCurrentSnapshot(employee.getId());
    Employee saved = employeeRepository.save(employee);
    String after = toJson(saved);
    auditService.writeAudit("admin", "UPDATE", "Employee", saved.getId(), before, after);
    return saved;
}
```

### 5.7. Як це пояснювати викладачу

> Журнал аудиту потрібний не для бізнес-логіки працівника, а для контролю змін у системі. Він зберігає інформацію про автора дії, тип дії, об'єкт зміни та старий і новий стан.

## 6. Патерн версіонування (Versioning Pattern)

### 6.1. Проблема, яку вирішує

Кадрова історія `JobHistory` фіксує лише організаційні події: прийняття, переведення, звільнення. Але вона не дає повної відповіді на питання:

- якою була вся картка працівника до зміни;
- хто саме змінив email, адресу, телефон;
- як виглядала сутність до помилки редагування.

### 6.2. Ідея патерна

При кожній зміні сутності створюється окрема версія її стану. Це дозволяє:

- бачити повну історію редагування;
- відновлювати старі стани;
- аналізувати еволюцію даних.

### 6.3. Логічна модель для Versioning Pattern

- `EmployeeVersion(id, employee_id, version_no, snapshot_json, changed_at, changed_by, change_reason)`

### 6.4. Важливе зауваження

У поточному HR System вже **частково реалізована ідея версіонування** через `JobHistory`, але тільки для кадрових подій. Повне версіонування є ширшим і покриває всю картку працівника.

### 6.5. Потік роботи

1. Користувач редагує працівника.
2. Перед оновленням зчитується поточний стан сутності.
3. У таблицю `EmployeeVersion` записується snapshot.
4. Основна таблиця `Employee` оновлюється.
5. За потреби стара версія може бути переглянута або відновлена.

### 6.6. Переваги

- повна історія змін сутності;
- простіше відкотити помилкове редагування;
- краща прозорість для аудиту.

### 6.7. Приклад коду

```java
public void createVersion(Employee employee, String changedBy, String reason) {
    EmployeeVersion version = new EmployeeVersion();
    version.setEmployeeId(employee.getId());
    version.setVersionNo(findNextVersion(employee.getId()));
    version.setSnapshotJson(toJson(employee));
    version.setChangedAt(LocalDateTime.now());
    version.setChangedBy(changedBy);
    version.setChangeReason(reason);
    employeeVersionRepository.save(version);
}
```

```java
@Transactional
public Employee updateEmployee(Employee form) {
    Employee current = findById(form.getId());
    versioningService.createVersion(current, "admin", "Редагування картки працівника");
    applyChanges(current, form);
    return employeeRepository.save(current);
}
```

### 6.8. Як це пояснювати викладачу

> Версіонування зберігає не тільки факт кадрової події, а повний стан сутності до зміни. Це дає можливість історичного аналізу і відновлення попередніх версій.

## 7. Порівняння патернів

| Патерн | Що вирішує | Де використовується | Який технічний артефакт доцільний |
| --- | --- | --- | --- |
| `Queue` | Асинхронна обробка подій | Події після зміни працівника, обробка документів, звіти | `QueueMessage`, worker |
| `Caching` | Прискорення читання | Довідники і звіти | Spring Cache / Redis / in-memory cache |
| `Audit Log` | Контроль дій користувачів | CRUD-операції, login, зміни статусів | `AuditLog` |
| `Versioning` | Історія станів сутності | Повна історія картки працівника | `EmployeeVersion` |

---

## 8. Підсумок для захисту

Для захисту курсової роботи цю частину краще формулювати так:

> У проєкті вже використані базові патерни `MVC`, `Repository`, `Service Layer`, `DTO/Projection`. Для підвищення надійності, масштабованості та контрольованості системи архітектурно обґрунтовано використання патернів `Queue`, `Caching`, `Audit Log` і `Versioning`. Вони узгоджуються з предметною областю HR System і логічно розширюють поточну модель БД.

Саме таке формулювання виглядає технічно сильним, не суперечить коду і демонструє інженерне розуміння архітектури системи.
